## O que ficou pronto

1. **Via parceira POR LOTE** — cada lote do edital é decidido sozinho (participar / via_parceira / descarte / revisão). Lote no escopo + trava de habilitação (exigir carta/vínculo de concessionária do licitante) = via parceira, vai a revisão pra humano confirmar a parceira. Nunca descarta edital que tem lote bom.
2. **Rollup blindado + 11 testes** automáticos garantindo o "0 descarte perigoso" no código
3. **Supabase recriado (setup v2)** — 8 tabelas + 6 views, regras corretas (ambulância de TODOS os tipos no escopo), prompt v5 idêntico ao código validado. Banco e Python não divergem: SQL é gerado do código (`gerar_sql_supabase.py`)
4. Extrator que lê a seção de habilitação funda no edital (onde mora a trava)

## Perguntas pra reunião (releitura de editais por erro no download)

1. **Que tipo de erro acontece?** Arquivo faltando (TR/anexo)? Versão errada (retificação)? Edital trocado? Info errada no repasse?
2. **Com que frequência?** Quantas releituras por semana/mês? (vira a baseline de retrabalho que o sistema vai zerar)
3. **Como o erro é descoberto e quanto custa?** Custeador percebe no meio da leitura? Quanto tempo perde por caso?
4. **O que define um "pacote completo"?** ⭐ A MAIS IMPORTANTE — lista exata do que o custeador precisa receber pra nunca voltar atrás (ED + TR + anexos + o quê mais?)
5. **2-3 exemplos concretos recentes** — pegar os PDFs (viram casos de teste do sistema)
6. **Retificações:** quem monitora hoje quando o edital muda depois de baixado? (se ninguém → vira o WF9)
